### Challenges Faced

### How We Resolved It
- **Interrupt model clarity:** Graph pauses at checkpoint (not killed), state persists to disk via checkpointer, resumes with one function call. This is HITL done right.

- **User input location:** Clarifications and enriched context go into `conversation_history` field in AgentState.
- **Key insight:** An unchecked agent optimizes for its metric alone (completeness) and becomes a runaway loop. Human oversight at critical decision points (like max_iterations) is necessary.

### Key Concepts & Student Understanding

**Agent Mental Model:**
- Agent = system that decides its next action based on observations (vs fixed pipeline/chain)
- Hybrid Deliberative-Reactive BDI Agent = our type
  - Deliberative layer: LLM thinks and plans
  - Reactive layer: tools execute fast responses
  - BDI: Beliefs (what it knows), Desires (goal), Intentions (committed actions)
  - Learning: stores failed queries, reflects (Reflexion pattern)

**PEAS Framework Applied:**
- Performance: ResearchBrief completeness, paper relevance, gaps identified, efficiency (≤4 iterations)
- Environment: ArXiv, Semantic Scholar, user knowledge (partially observable, stochastic, sequential)
- Actuators: arxiv_search, semantic_scholar_fetch, paper_ranker, gap_analyzer, ask_questions, synthesize
- Sensors: paper metadata, citation counts, API errors, user answers, own state

**Agent Loop:**
- Perceive (read state + observations) → Think (LLM decides) → Act (call tool) → Observe (get result) → Check goal met? → Loop or terminate

**Three Lines That Define an Agent (the "soul"):**
1. Adaptive routing: `should_continue()` decides next node based on observations (not fixed path)
2. Persistent state: `AgentState` carries beliefs, desires, intentions across nodes
3. Tool-grounded reasoning: every thought checked against reality (tool results), not hallucinated

**HITL Architecture (Interrupt Model):**
- Graph pauses cleanly at checkpoint (not killed) via `interrupt_after=["node_name"]`
- State persists to disk via checkpointer (e.g., SqliteSaver) with thread_id as key
- Can wait indefinitely for user input
- Resume with `graph.update_state()` + `graph.invoke(None, config)` — no restart, full context intact
- Better than shutdown model (process death) because state is preserved

**Termination & User Decision Points:**
- At `iteration_count >= max_iterations`: agent does NOT keep searching (would be runaway loop)
- Instead: pause, show state (papers, gaps, history), ask user "continue 3 more or wrap up?"
- User can provide enriched context ("I'm interested in vision transformers specifically")
- This enriched context goes into `conversation_history` field for next iteration
- Key insight: unchecked agent optimizes metric alone (completeness) and ignores real constraints (time, cost). Human oversight at critical junctures is essential.

### ReAct Misconception Clarified
- Student initially thought ReAct required "thinking capability" model (like o1/extended thinking)
- **Reality:** ReAct is pure **code choreography + prompting** (not a model feature)
- **How it works:** Prompt tells model to follow format (Thought/Action/Observation). Your code:
  1. Sends prompt to LLM
  2. Parses "Action: {tool_json}" from response
  3. Executes actual tool
  4. Formats result as "Observation: [result]"
  5. Feeds back into prompt loop
  6. Model continues (unaware tools are real—just following format)
- **Key insight:** Model follows prompt format. Code choreographs the loop. Not a model capability.

## TOOL SPECIFICATIONS (Locked In)

### Tool #1: arxiv_search_tool
- **Input:** query (str), max_results (int), date_range_start (optional str), date_range_end (optional str)
- **Output:** List[PaperMetadata]
- **Source:** ArXiv API (free, no key needed)
- **Error handling:** Return structured error string if timeout or 0 results, never raise exception

### Tool #2: semantic_scholar_tool
- **Input:** arxiv_id (str) — NOT paper name
- **Output:** CitationData (citation_count, influential_citation_count, h_index)
- **Source:** Semantic Scholar API (free)
- **Error handling:** Return empty CitationData if API fails, never raise exception

### Tool #3: paper_ranker_tool
- **Input:** papers (List[RankedPaper]), weights (dict with citation_weight, relevance_weight, recency_weight)
- **Output:** List[RankedPaper] sorted by composite_rank_score (descending), with rank_position set
- **Source:** Pure calculation (no external API)
- **Formula:** composite_score = (citation_norm × citation_weight) + (relevance_score × relevance_weight) + (recency_score × recency_weight)

### Tool #4: gap_analyzer_tool
- **Input:** papers (List[RankedPaper]), goal (str), conversation_history (List[dict])
- **Output:** List[KnowledgeGap]
- **Source:** LLM call inside the tool (structured output via with_structured_output())
- **Purpose:** Read collected papers, identify what we still don't know about the goal

### What We Built

### What the Student Got Right
- Minimum required initial state: `goal`, `max_iterations`, `status` (dropped iteration_count correctly)
- `thread_id` as the checkpointer session key — not the checkpointer object itself
- `astream()` vs `invoke()` distinction: visibility into intermediate state, not parallel processing
- Correctly identified `ranking_node` as the source of the paper duplication bug

### Bugs Found in First Run
1. **ArXiv 429 rate limit** — fired 5 parallel queries too fast → tool fell back to mock papers (expected, not a code bug)
2. **Reducer duplication bug** — `ranking_node` returns `{"papers": ranked}` → `operator.add` appends ranked list ON TOP of collected list → 20 papers instead of 10
3. **LangGraph deserialization warning** — `RankedPaper`, `KnowledgeGap`, `ResearchStrategy` are unregistered types in checkpointer msgpack. Will become a blocking error in future LangGraph versions. Fix: register types or add to `allowed_msgpack_modules`.

### How We Resolved
- **Duplication fix (deferred):** Move ranking inside `paper_collection_node`, rank before returning to state. `ranking_node` should not return `papers` field at all.
- **Deserialization warning:** Noted — fix in MODULE 9 (context budgeting + state management)
- **Rate limit:** Add small delay between parallel ArXiv queries in future iteration

### Key Insights from Session 4
- `astream()` gives per-node visibility — caught the duplication bug immediately from output numbers
- `operator.add` reducer is correct for accumulating across *iterations*, but wrong when the same node re-adds already-stored data within one iteration
- `get_state(config)` after stream ends retrieves full final state from checkpointer — this is the HITL resume backbone
- `invoke()` loses intermediate visibility; for long-running agents (30-60s), this feels like a frozen app
